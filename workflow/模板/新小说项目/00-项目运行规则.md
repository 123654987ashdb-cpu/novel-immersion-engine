# {{PROJECT_NAME}} - 项目运行规则

本项目使用“小说沉入工程”工作流。若你是正在接手本项目的 AI，即使你没有安装对应技能，也必须先遵守本文件，再读取 `00-项目控制台.md` 进行工作；若已安装 `$novel-immersion-engine`，应使用它作为本项目的策划驾驶技能。若已安装 `$novel-prose-renderer`，可用它作为正文执行技能；若已安装 `$novel-revision-learning-loop`，可用它作为作者改稿学习与经验打包技能。

## 技能与项目的关系

- 本项目文件夹是作品的持久状态与正式产物所在地。
- `$novel-immersion-engine` 负责读取本项目、完成推理、生成/更新文件和推动节点，不建立一套脱离项目的聊天版状态。
- `$novel-prose-renderer` 负责把《阶段正文交接包》写成小说正文，不重新策划剧情，不替换固定锚点。
- `$novel-revision-learning-loop` 负责比较蓝图、AI 初稿和作者终稿，提炼可回流的总体经验与文笔经验。
- 总纲、分卷/阶段/章节规划、交接包、验收结论、状态变化和复盘，只要属于本项目，就必须写回本项目对应目录。
- 作者改稿复盘、经验增量和经验包，只要属于本项目，也必须写回 `11-作者改稿学习与经验包`。
- 聊天回复用于汇报已完成文件、待作者选择与下一步，不代替正式项目文件。

## 职责分工

- 策划端：处理结构、因果、人物与关系变化、伏笔、信息边界、物品、技能/物品效果、代价、节奏、状态更新和正文验收。
- 正文端：依据《阶段正文交接包》和可用的《文笔执行经验包》写正文，在边界内发挥文字与场景创造力。
- 学习端：依据蓝图、AI 初稿、作者终稿和已接受正文，复盘作者为什么改、改成了什么、哪些经验能进入下一轮。
- 作者：选择总纲与重大改线，接受或否决关键正文发现。

默认情况下，策划端不直接生成正文。

## 五级推演

按以下顺序推进，不从零散想法直接跳到正文：

```text
小说圣经 -> 小说总纲 -> 分卷纲 -> 卷内阶段纲 -> 章节纲 -> 内部执行包/场景卡 -> 阶段正文交接包
```

- 总纲负责整本书的闭环、大伏笔、主角和关系终点。
- 分卷纲负责每卷的矛盾、回报、高潮和阶段转变。
- 阶段纲负责连续若干章的起止状态、因果链、回报和节奏。
- 章节纲负责每章的功能、状态改变、伏笔动作和结尾牵引。
- 内部执行包/场景卡负责在策划端解决认知、债务、物品、能力边界、代价和关键转向。
- 交接包负责将本章必须成立的内容与允许正文创造的内容分开。

## 权威顺序

从高到低：

1. 已接受正文与作者明确确认的 canon。
2. 作者明确的长期意图、结局重力和禁止方向。
3. 当前阶段/当前章节的局部指令。
4. 已选但尚未写成正文的未来规划。
5. 候选方案、推测和正文草稿中的未确认发现。

不得让低优先级内容偷偷覆盖高优先级内容。

## 创造自由

正文交接包必须区分：

| 类型 | 处理方式 |
| --- | --- |
| 固定锚点 | 必须实现，不得改变结果 |
| 边界内创造 | 目标和边界固定，正文端自行发明具体过程 |
| 开放创造位 | 只规定需产生的读者感受或戏剧功能，不规定台词、动作和意象 |

正文端写出的局部惊喜，只要不新增长线负债、不改变关键状态，不应被视为错误；验收时可保留为有效发现。

## 正文回流

1. 交接包保存于 `05-正文交接包`。
   - 复杂整章、续写或有关键状态变化时，使用 `完整交接包模板.md`。
   - 只有低风险短场景、试写或局部补写，才可使用 `交接包模板.md` 快速版。
2. 正文模型返回的稿件先进入 `06-待验收正文`，不能直接进入 canon。
3. 策划端在 `08-验收与状态增量` 逐章验收。
4. 通过后的正文进入 `07-已接受正文`，并更新 `04-当前故事状态` 中启用的账本。
5. 一个阶段写完后必须在 `09-阶段复盘与校准` 复盘；最多不超过两个相邻阶段。
6. 一卷完成后使用 `卷末复核模板.md`，结算本卷债务并明确下卷继承。

## 作者改稿学习闭环

作者的修改不是终点，而是训练信号。每当存在“章节蓝图 / AI 初稿 / 作者终稿”三份材料时，可以启动学习闭环：

```text
阶段正文交接包
  -> 正文端生成 AI 初稿
  -> 作者修改成终稿
  -> 学习端对照蓝图、AI 初稿、作者终稿
  -> 生成作者改稿复盘
  -> 拆分为总体经验增量与文笔经验增量
  -> 多次积累后打包为整体创作经验包与文笔执行经验包
```

- 原始复盘可包含本书具体内容，只保存在本项目内部。
- 经验增量用于说明本次改稿能给策划端或正文端带来什么教训。
- 经验包必须去内容化：不泄露角色名、专属设定、关键剧情和原文，只保留可迁移方法。
- `整体创作经验包` 给策划端读取，影响后续总纲、阶段、章节蓝图和交接包。
- `文笔执行经验包` 给正文端读取，影响下一次正文的场景、对话、情绪、节奏和句子。
- 作者一次修改不自动成为永久规则；学习端必须标明适用范围与置信度。

## 项目文件规则

- 先读本运行规则和控制台，再按当前动作加载必要材料。
- 候选内容不写进确定事实。
- 各类账本按 `04-当前故事状态/账本启用清单.md` 启用，不要求无关项目机械填表。
- 每份整章交接包之前，策划端先完成所需的内部执行包与场景卡；这些后台材料不原样塞给正文模型。
- 策划端制作交接包时，应读取 `整体创作经验包` 中适用于当前项目的经验；正文端写作时，应读取 `文笔执行经验包` 中适用于当前章的经验。
- 学习端生成经验包时，应区分原始复盘、经验增量和去内容化经验包，不得把本书私有内容塞进通用经验。
- 每完成一项正式产物，更新控制台：当前节点、最新文件、待作者决定与下一步。
