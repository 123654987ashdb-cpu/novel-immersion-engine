# 小说沉入工程：与 GPT / Claude 配合使用

在 Codex / Claude Code 中，`小说沉入工程` 技能现在默认作为本地工作流项目的驾驶技能：规划、状态与验收记录应落在项目文件中。普通 GPT / Claude 默认仍是正文端；若你将项目材料上传给策划 GPT，它应遵循同一套项目状态与交接规则。

这套系统的默认分工是：

- **Codex 或 Claude Code + `$novel-immersion-engine`**：负责按五级结构推理、设计、维护项目状态，并输出阶段导航与《章节写作蓝图》。
- **GPT 或普通 Claude 对话 + `$novel-prose-renderer` 或正文执行提示**：负责根据蓝图生成正文，发挥文笔，并吸收《文笔执行经验包》。
- **你**：判断正文是否成立，必要时亲手改成作者终稿。
- **Codex / Claude Code / GPT + `$novel-revision-learning-loop`**：复盘“AI 初稿 -> 作者终稿”的变化，沉淀《整体创作经验包》和《文笔执行经验包》。
- **Codex 或 Claude Code**：接收正文回来验收；一个阶段完成后，或最迟连续两个阶段完成后，做一次阶段复盘，更新故事状态并校准还没写出的后续阶段。

推理不能孤立地只看一章：策划端先完成 `小说总纲 -> 分卷纲 -> 每卷 3-4 个阶段纲 -> 章节纲 -> 章节细纲/写作蓝图`。不要把完整小说圣经和全部账本都塞给正文模型；让它只看到当前阶段导航和目标章蓝图。

这里的蓝图不是填空题。策划端锁住的是必须成立的剧情锚点、信息边界和后续不能被破坏的状态；正文端必须拥有真实的创作空间，自行找到更有生命力的场景路径、潜台词、即时阻碍、人物反应、意象回声和局部惊喜。

## 推荐流程

1. 在 Codex 或 Claude Code 中建立小说圣经，并先推理、选择整本书的小说总纲。
2. 沿总纲展开分卷纲；每卷拆成通常 3-4 个连续阶段纲；阶段通过后再分配章节纲。
3. 需要生成正文时，让策划端输出《阶段正文交接包》：紧凑的当前阶段导航，加当前章或指定小批章节的《章节写作蓝图》。
4. 将交接包完整粘贴给 GPT 或 Claude，并附上下面的“正文模型执行提示词”；如果有《文笔执行经验包》，一并附上。
5. 你阅读 AI 初稿：可以直接接受，也可以亲手改成作者终稿。
6. 如果你改过正文，用 `$novel-revision-learning-loop` 对照交接包、AI 初稿和作者终稿，提炼总体经验增量与文笔经验增量。
7. 将最终正文交回策划端逐章验收；通过的正文才进入确定事实。
8. 当一个阶段写完，或连续写完两个阶段时，把这些已验收正文交给策划端做《已完成阶段复盘与后续校准》。
9. 策划端检查本阶段是否真正完成任务、正文是否发现更好的变化，并只对还没写出的阶段纲和章节蓝图进行修正，再输出下一份交接包。

## 写完一段或两段后怎么办

不需要死守最初的整卷细节，也不能让正文模型自己改路线。正确做法是：

1. GPT 或 Claude 按当前《阶段正文交接包》生成正文。
2. 你改稿或确认终稿；改过的内容先触发作者改稿学习，而不是只留下一个“最终版”。
3. Codex 或 Claude Code 再逐章验收，把确认通过的内容写入当前事实。
4. 一个阶段完成后进行阶段复盘；若一次连续生成较多内容，最迟不能超过两个阶段再复盘。
5. 复盘时对照原阶段任务，查看读者回报、人物和感情变化、伏笔、物品、技能/物品效果、代价、节奏是否落实，以及正文中是否出现值得保留的新细节。
6. 已接受的正文优先于旧计划；尚未写出的阶段可以微调。若影响本卷高潮或全书方向，则由策划端给出改线方案，由你选择后再继续写。

这一步的作用，就是让骨架不是死的：它能控制长线，又能吸收真正写出来以后更好的东西。

## 正文模型执行提示词

将下面提示词放在本章蓝图前面使用：

```text
请使用 $novel-prose-renderer。如果当前环境没有这个技能，就严格按下面的正文执行规则工作。

你现在只负责小说正文表现，不负责重新策划剧情。

我将提供一份《阶段正文交接包》，其中包含当前阶段的连续章节导航，以及目标章的《章节写作蓝图》。阶段走向和蓝图中的固定锚点、场景因果、人物关键选择、关系变化、信息边界、伏笔功能、技能/物品效果边界、物品/资源状态、核心代价变化和结尾落点已经由策划端确定，你必须遵守。

如果我同时提供《文笔执行经验包》，你只能把它当成表达经验使用：学习它如何处理句子、节奏、对白、潜台词、感官、留白和情绪落点，不得把其中的示例内容当成当前小说事实。

你不是把提纲机械扩写成句子。对于蓝图标明的“边界内创造”和“开放创造位”，请主动发挥想象力，寻找最自然、最有感染力或最有惊喜感的执行方式。你可以自由创造：抵达固定结果的具体场景路径、人物临场战术、即时小阻碍、动作细节、潜台词、场景氛围、感官描写、意象回声、自然过渡、符合人物状态的对白表达、节奏与文字质感。

你不得擅自：
1. 新增需要后续承担后果的重要设定或伏笔（不限制不造成后续负债的生活化细节和局部发现）；
2. 改变人物立场、决定或关系转折；
3. 提前揭露蓝图要求隐藏的信息；
4. 临时新增、升级或越界使用技能/物品效果，或改变关键物品、资源、核心代价的既定状态；
5. 把主题结论直接塞进角色台词；
6. 改写既定结尾方向。

请只输出完整本章正文，不输出分析、解释、提纲或写作说明。
如果蓝图内部存在无法同时满足的明确矛盾，请停止生成正文，只列出矛盾点。

以下是阶段正文交接包：
```

随后粘贴策划端生成的《阶段正文交接包》。

## 必要时可附带什么

通常只需要阶段导航和蓝图。遇到下列情况，可以额外附少量材料：

- 需要无缝续接时：附上一章结尾约 500-1500 字。
- 需要稳定人物口吻时：附该人物少量已确认对白片段。
- 需要稳定叙述气质时：附极短的声音约束或一小段已接受正文。
- 已经积累过作者改稿经验时：附《文笔执行经验包》中与本章有关的少量条目。

不要额外上传完整账本，除非你希望 GPT 同时承担策划职责。

## 作者改稿后怎么让系统学习

当你把 AI 初稿改成更满意的终稿后，把三份材料交给学习端：本章交接包、AI 初稿、作者终稿。推荐提示词：

```text
请使用 $novel-revision-learning-loop 做作者改稿复盘。
请对照本章《阶段正文交接包》、AI 初稿、作者终稿和我的接受判断，分析我改动的真实目的。

请输出：
1. 作者改稿复盘；
2. 总体经验增量：以后策划端怎样在大纲、细纲、蓝图中提前安排；
3. 文笔经验增量：以后正文端怎样在句子、节奏、对白、潜台词、留白、感官和情绪上执行；
4. 若已经有多次复盘，请更新去内容化的《整体创作经验包》和《文笔执行经验包》。

经验包不要保留本小说的私有人名、剧情、设定和原句，只保留可迁移的写作方法。
```

## 把 GPT 当正文端时

如果你想给正文端单独做一个自定义 GPT，上传：

- `novel-prose-renderer/SKILL.md`
- `novel-prose-renderer/references/prose-execution.md`
- `novel-prose-renderer/references/prose-experience-pack.md`

它的职责只有一个：根据《阶段正文交接包》和《文笔执行经验包》写小说正文，不重新策划剧情，不输出分析说明。

## 把 GPT 当学习端时

如果你想给复盘学习单独做一个自定义 GPT，上传：

- `novel-revision-learning-loop/SKILL.md`
- `novel-revision-learning-loop/references/revision-learning-format.md`
- `novel-revision-learning-loop/references/experience-packaging.md`

它的职责不是写正文，而是从“AI 初稿与作者终稿的差异”中提炼方法，分别喂给策划端和正文端。

- `skill/novel-immersion-engine/SKILL.md`
- `skill/novel-immersion-engine/references/story-bible-and-ledgers.md`
- `skill/novel-immersion-engine/references/hierarchical-outline-system.md`
- `skill/novel-immersion-engine/references/chapter-scene-system.md`

- `skill/novel-immersion-engine/references/variants-and-character-evolution.md`
- `skill/novel-immersion-engine/references/project-state-and-production.md`
- `skill/novel-immersion-engine/references/serialization-governance.md`

为这个策划 GPT 使用下面指令：

```text
你是“小说沉入工程”策划端。你的职责不是默认生成正文，而是在正文生成之前解决结构、因果、人物变化、关系变化、爽点/痛点、伏笔回收、信息边界、关键物品、技能/物品效果、资源、核心代价、节奏与连续性问题。

你的推理必须按五级结构推进：先生成并比较小说总纲方案，选定后检查整本书闭环；再展开分卷纲；每一卷通常拆成 3-4 个卷内阶段纲；阶段成立后才生成章节纲；最后将需要写正文的章节展开为章节细纲/《章节写作蓝图》。大纲负责大约发生的事和大伏笔，细纲负责场景事件、细微伏笔、具体感情变化与正文执行约束。

你必须维护确认过的故事状态，将候选想法与 canon 区分开。需要写正文时，输出《阶段正文交接包》：先提供当前阶段及相邻章节的紧凑导航，再提供目标章的一份或少量几份《章节写作蓝图》。蓝图必须清楚写出本章功能、开始状态、人物执行状态、场景级责任、必须落地、固定锚点、边界内创造、开放创造位、禁止越界、文笔与想象力发挥空间和输出要求。不要把完整后台账本倾倒给正文模型，也不要把正文端能创造的每个动作和瞬间预写死。

当我把 GPT 或 Claude 生成的正文交回来时，你负责按蓝图与 canon 逐章验收；只有通过或我明确接受的状态变化才写回当前故事状态。当一个阶段完成，或最多两个相邻阶段已完成时，你必须做一次阶段级复盘，对照原阶段目标评估实际成效，并校准尚未写出的后续阶段纲和蓝图；已接受正文高于旧计划，重大改线先交给我选择。

除非我明确要求你试写正文，否则你输出阶段正文交接包，不输出小说正文。
```

## 情感代价型故事扩展

当一部小说以“主角正在逐步失去某种重要部分”为留存机制，并且存在记录物或见证角色时，可参考 `project-overlay-example.md`，向策划端追加：

```text
请参考 project-overlay-example.md，为本项目建立专属的累积代价与见证记录层。先依据已有正文或我的明确决定，确认可测指标、人物损耗、记录物、见证作用和调查者认知风险中哪些已属于 canon，哪些仍是候选。在总纲、分卷纲和阶段纲中安排这条损耗线的长期与阶段进展；生成阶段正文交接包时，将目标章真正需要的损耗表现、记录动作、角色认知边界和留存牵引提炼进去，不要把整个项目账本交给正文模型。主题必须通过损耗、反应延迟、记录、选择和关系变化呈现，不能让角色代替作者总结。
```
