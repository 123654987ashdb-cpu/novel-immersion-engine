# 小说沉入工程：完整使用说明

> 公开仓库入口见 [README.md](../README.md)。新建项目和长期创作时，从 [workflow/开始这里.md](../workflow/开始这里.md) 开始：**工作流保存小说项目，`$novel-immersion-engine` 技能负责接管并驾驶它。** 本说明保留为系统规则与协作方式的完整参考。

## 1. 这套系统是干什么的

`小说沉入工程` 不是让 AI 直接凭感觉写长篇小说的提示词，也不是单纯的大纲模板。正式长期创作中，它是使用本地工作流项目的策划驾驶技能：结果要写入项目文件，而不是只留在聊天里。

它的作用是把小说创作拆成两类工作：

- **需要推理和控制的工作**：故事结构、人物改变、感情推进、伏笔回收、信息边界、关键物品、技能/物品效果、代价、节奏、连续性和后续改线。
- **需要文字表现的工作**：场景呈现、动作细节、对白质感、氛围、节奏和文笔。

默认分工如下：

| 角色 | 适合使用的模型 | 负责什么 |
| --- | --- | --- |
| 策划端 | Codex / Claude Code / 配置好的策划 GPT + `$novel-immersion-engine` | 推演全书和当前阶段，输出正文交接包，验收正文，校准后路，并读取整体创作经验包 |
| 正文端 | GPT / Claude 对话 + `$novel-prose-renderer` 或正文执行提示 | 按交接包写出正文，发挥语言表现力，并吸收文笔执行经验包 |
| 学习端 | Codex / Claude Code / GPT + `$novel-revision-learning-loop` | 对照蓝图、AI 初稿和作者终稿，提炼总体经验与文笔经验 |
| 作者 | 你 | 选择方向、判断喜欢与否、决定重大改线和最终接受正文 |

一句话理解：

> 策划端先把故事的骨架、因果和变化想透；正文端负责把它写得好看；作者改完以后，学习端把“为什么这样改更好”沉淀成经验，再反哺下一轮策划和正文。

这不意味着正文端只是填空。系统要锁住的是会破坏长篇的关键错误，不是提前消灭场景里的想象力。

## 2. 最重要的工作流

长篇小说按五级结构逐步向下推演：

```text
小说圣经
  -> 小说总纲
  -> 分卷纲
  -> 卷内阶段纲（每卷通常 3-4 段）
  -> 章节纲
  -> 章节写作蓝图 / 阶段正文交接包
  -> GPT 或 Claude 使用正文技能/提示生成正文
  -> 作者修改或确认正文
  -> 作者改稿学习复盘（可选但强烈推荐）
  -> Codex 或 Claude Code 逐章验收
  -> 阶段完成后复盘并校准后续路线
```

其中：

| 层级 | 主要解决什么 |
| --- | --- |
| 小说总纲 | 整本书的核心冲突、结局方向、人物终点、大伏笔和分卷路线 |
| 分卷纲 | 本卷的矛盾、高潮、主要回报、本卷人物/关系变化 |
| 卷内阶段纲 | 连续若干章怎样把状态从 A 推到 B，压力和回报如何分布 |
| 章节纲 | 每章大约发生什么、本章承担什么功能、留下什么牵引 |
| 章节写作蓝图 | 场景级事件、细微情绪、伏笔动作、不能越界的内容和文笔发挥空间 |

不要一上来就让模型写第一章。先让它知道这本书往哪里走、当前这一段为什么必须发生，再写正文，稳定性会高很多。

同时，不要把蓝图写成逐句执行说明。一个好的蓝图会区分：

| 内容 | 是否锁定 |
| --- | --- |
| 固定锚点：关键选择、揭示、关系结果、代价、结尾 | 锁定，不能乱改 |
| 边界内创造：冲突怎样升级、人物如何周旋、怎样表现情绪 | 交给正文模型创造 |
| 开放创造位：只规定需要产生的触动或不安，不规定具体做法 | 主动留白，等待惊喜 |

## 3. 在哪里使用

### 3.1 在 Codex 中使用

技能已经安装。长期写作时，先通过工作流创建并打开一本小说项目，再让技能接管它：

```text
请使用 $novel-immersion-engine 接管本小说项目。
先读取《00-项目运行规则.md》《00-项目控制台.md》和《01-输入素材》，按控制台的下一步推进。
不要默认直接写正文；把正式产物写回项目对应目录并更新控制台。若当前需要选择方向，请给我 3 个真正不同的方案。
```

界面显示名称是“小说沉入工程”，内部调用名保留为 `$novel-immersion-engine`。

### 3.2 在 Claude Code 中使用

技能已经安装。打开工作流中的具体小说项目后，可以这样开始：

```text
请使用 novel-immersion-engine 技能接管本小说项目。
先读取《00-项目运行规则.md》和《00-项目控制台.md》，按当前节点完成产物并写回项目；不默认直接写正文。
```

### 3.3 在普通 GPT 或 Claude 中使用

普通 GPT 或 Claude 更适合作为**正文端**。策划端给出《阶段正文交接包》后，把交接包发给它，让它只写正文。

如果你希望 ChatGPT 自身也承担策划端，可按照 [GPT与Claude正文端使用.md](GPT与Claude正文端使用.md) 上传技能文件并配置策划指令。

## 4. 你从哪一种情况开始

以下三种路线用于正式长期创作时，都先在工作流中创建小说项目，把素材放入 `01-输入素材`，再让 `$novel-immersion-engine` 接管项目。若只是临时讨论想法，可不建立项目，但结果仍是候选内容。

### 路线 A：你只有一个小说想法

适合：刚有题材、设定、人物关系或一个很想写的画面，还没有正式大纲。

第一条指令：

```text
请使用 $novel-immersion-engine 接管本小说项目。我已将初始想法放入《01-输入素材》。先读取《00-项目运行规则.md》《00-项目控制台.md》和输入素材，暂时不要写正文。

请先完成：
1. 整理小说圣经：题材、核心读者体验、故事钩子、主角欲望与缺口、主要阻力、核心代价、结局气质。
2. 给出 3 个真正不同的小说总纲方向，不要只是换措辞。
3. 对比三个方向的爽点/情绪牵引、人物改变、关系变化、伏笔空间、长篇风险和结局力度。
4. 等我选择后，再展开分卷纲。

请将结果写入项目对应目录并更新控制台，不要只在聊天中输出正式产物。
```

你需要决定的是：哪一个方向最想追下去，哪些部分要混合，哪些坚决不要。

### 路线 B：你已经有自己的大纲

适合：你已经有故事主线、分卷构想、人物关系或结局，只希望 AI 帮你增强和落地。

第一条指令：

```text
请使用 $novel-immersion-engine 接管本小说项目。我已将自己的大纲放入《01-输入素材》。先读取项目规则、控制台和素材。

请保留我真正想写的核心，不要直接替换成另一部模板化小说。先判断我的大纲目前具备哪些层级：总纲、分卷纲、阶段纲、章节纲；再指出缺失、薄弱或会导致正文失控的地方。

请输出：
1. 我的大纲核心不可丢失的部分；
2. 高影响问题，按严重程度排序；
3. 2-3 个加强方向及代价；
4. 我选择后，再把选定方向修成可继续展开的正式大纲。

请将诊断与候选方案写入项目对应目录并更新控制台，不要把候选内容写入 canon。
```

### 路线 C：你已经写了一部分正文，需要续写

适合：已有若干章正文，可能有旧大纲，也可能只有已写文本。

第一条指令：

```text
请使用 $novel-immersion-engine 接管本小说项目。我已将已有正文放入《01-输入素材》，需要在不破坏既有内容的前提下继续规划。

请不要马上续写正文。先：
1. 按章把已有正文反向整理为章节纲；
2. 提取已经确定的事实、人物状态、关系状态、伏笔、信息边界、关键物品、技能/物品效果、核心代价和未解决问题；
3. 判断故事目前位于总纲、分卷纲和阶段纲中的什么位置；
4. 发现旧大纲与正文冲突时，以已经接受的正文为优先，并把冲突告诉我；
5. 给出后续路线方案，确认后再生成下一阶段的正文交接包。

请将反向整理、当前状态和后续候选路线写入项目对应目录并更新控制台。
```

如果正文较长，建议每次处理 5-10 章，先把状态稳定整理出来，再向后规划。

## 5. 从构思到正文的实际步骤

### 第一步：建立小说圣经并选择总纲

策划端会整理故事引擎、人物初始状态、读者想追的东西和结局气质，并把多个总纲方向写入项目的总纲候选文件。

你回复时可以非常直接：

```text
我选择方案 B，但保留方案 A 的人物关系设计；不要方案 B 中的某个设定。请在本项目中据此形成正式小说总纲，检查结局、人物变化和大伏笔能不能闭环，并更新控制台。
```

此阶段不要急于设计每一章。先确认整本书值得写。

### 第二步：展开分卷纲

当总纲选定后，要求策划端给出每一卷负责解决的问题。

```text
总纲方向确定。请在本项目中展开分卷纲，并将正式产物保存到规划目录、更新控制台。
要求每一卷明确：开始状态、核心矛盾、主要回报、高潮、人物/关系变化、主要伏笔动作、卷末留下的问题。
不要先写章节细纲。
```

你需要重点看：

- 每一卷是不是都改变了故事状态。
- 是否有一卷只是过渡，没有真正回报。
- 人物变化是不是被分配到了过程中，而不是最后突然改变。

### 第三步：把当前卷拆成阶段纲

每卷通常拆成 3-4 个连续阶段，不要求机械平均。

```text
请在本项目中将第 X 卷拆成 3-4 个卷内阶段纲，保存产物并更新控制台。
每个阶段说明：开始状态、结束状态、连续因果链、主要回报、人物/感情变化、伏笔任务、预计覆盖章节，以及下一阶段必须继承的压力。
```

这是整个系统最关键的一步。它防止模型只会写眼前一章，却不知道连续几章如何形成一股推进力。

### 第四步：生成当前阶段的章节纲

选定当前阶段后，再分配章节功能。

```text
我选择第 X 卷阶段 Y。请在本项目中生成这个阶段的连续章节纲，并保存产物、更新控制台。
每章标明：本章功能、大约事件、主要冲突或选择、读者回报、人物/感情推进、伏笔动作、状态改变和结尾牵引。
先不要生成正文。
```

### 第五步：生成《阶段正文交接包》

真正准备写正文时，让策划端将当前阶段中要写的章整理成可交给 GPT 或 Claude 的材料。

```text
请使用 $novel-immersion-engine 为本项目第 X 卷阶段 Y 的第 A 章生成《阶段正文交接包》，保存到 `05-正文交接包` 并更新控制台。
交接包要包含紧凑的阶段导航、相邻章节骨架和本章完整《章节写作蓝图》。
蓝图必须区分固定锚点、边界内创造和开放创造位；不要把正文端能发挥的场景过程全部预写死。
不要写正文，不要把完整后台账本发给正文端。
```

建议第一次先只做一章。等你确认正文端确实能听懂交接包，再一次制作 2-3 章的交接材料。

## 6. 怎样让 GPT 或 Claude 写正文

把策划端输出的《阶段正文交接包》完整发给正文端，并在最前面加上下面这段：

```text
请使用 $novel-prose-renderer。如果当前环境没有这个技能，就严格按下面的正文执行规则工作。

你现在只负责小说正文表现，不负责重新策划剧情。

我将提供一份《阶段正文交接包》，其中包含当前阶段导航和目标章的《章节写作蓝图》。阶段走向、固定锚点、人物关键选择、关系变化、信息边界、伏笔功能、技能/物品效果边界、物品/资源状态、核心代价变化和结尾落点都已经确定，你必须遵守。

如果我同时提供《文笔执行经验包》，你只能把它当成表达经验使用：学习它如何处理句子、节奏、对白、潜台词、感官、留白和情绪落点，不得把其中的示例内容当成当前小说事实。

你不是把提纲机械扩写成正文。对于蓝图标注的边界内创造与开放创造位，请主动发挥想象力，自行设计抵达固定结果的最好路径，包括合理的小阻碍、临场动作、潜台词、人物反应、意象回声、感官体验、自然对白、过渡、节奏与文字质感。

你不得擅自新增需要后续偿还的重要设定或伏笔，不得临时新增、升级或越界使用技能/物品效果，不得改变人物关键决定和关系转折，不得提前揭露隐藏信息，不得改变关键物品、资源、核心代价或既定结尾，也不得让角色直接说出作者想表达的主题结论。不造成后续负债且符合人物和世界的局部创作发现是允许的。

请只输出完整正文，不输出分析、提纲或写作说明。如果交接包内部有明确矛盾导致无法同时满足，请停止写作并只指出矛盾。

以下是《阶段正文交接包》：
```

随后粘贴交接包。

### 正文端可以额外收到的材料

| 情况 | 可以额外附带什么 |
| --- | --- |
| 需要紧接上一章 | 上一章结尾约 500-1500 字 |
| 人物说话容易失真 | 该人物少量已经确认的对白 |
| 文风明显漂移 | 一小段已经接受的正文或极短声音要求 |

不要把整本小说圣经、所有账本和全部未来大纲都塞给正文端。它需要知道怎么写当前内容，不需要重新承担整部小说的推理。

## 7. 怎样避免蓝图把小说写死

蓝图的作用是让正文端不要在关键问题上乱决定，而不是让它失去创造欲。实际使用中，坚持四条原则：

1. 只锁住会影响因果、人物、伏笔、代价和后续路线的内容。
2. 只要多种写法都能实现本章目标，就把具体实现方式留给正文端。
3. 每章尽量设置至少一个开放创造位，例如“让她第一次察觉他正在改变，但不允许她确认原因”，而不是预先指定每个动作和每句台词。
4. 正文写出更妙的局部细节时，先验收它是否破坏长线；不破坏就可以保留，并在阶段复盘时决定它是否值得进入后续设计。

例子：

| 过度写死 | 更好的交接方式 |
| --- | --- |
| “她看见他手抖，问你怎么了，他说没事，她把杯子放下。” | “她必须第一次感到他的日常反应有异常；她尚不能确认原因。开放创造具体触发物、停顿和试探方式。” |
| “必须用雨声表现疏离。” | “本场结束时关系靠近失败，并留下冷却感；意象和动作由正文端选择。” |
| “他必须用某句话羞辱对方。” | “他必须公开压住对方并暴露自己的控制欲；语言战术由正文端设计。” |

好的交接包应该像一座有方向的舞台，不是一张把演员手脚都绑住的动作清单。

## 8. 正文写出来后怎么处理

正文生成后，不要直接把它当作确定版本继续往下写。在工作流项目中，把正文放入 `06-待验收正文`，再交回策划端：

```text
请使用 $novel-immersion-engine 验收 `06-待验收正文` 中根据交接包生成的第 X 章正文。读取对应交接包、内部执行材料、当前状态和已启用账本。

请检查：
1. 是否完成了蓝图要求的事件、回报、人物和关系变化、结尾牵引；
2. 是否出现信息越界、人物突然变性、伏笔提前暴露、技能或物品效果越界/无铺垫升级、关键物品/资源/核心代价错误；
3. 是否有值得保留的自然细节或表达发现；
4. 哪些内容可以进入确定事实，哪些需要修改或由我决定；
5. 通过后请将本章状态增量与下一章必须继承内容写回项目，并更新控制台。
```

策划端会把正文分成四类处理：

| 情况 | 处理方式 |
| --- | --- |
| 按蓝图很好地完成 | 接受正文，并更新状态 |
| 有漂亮的非关键发挥 | 建议保留，写入风格或细节记录 |
| 有局部偏移但可修 | 给出修改点，修好后再接受 |
| 改变关键路线 | 不自动接受，交给你选择 |

如果你不是直接接受 AI 初稿，而是亲手改成了更满意的版本，这一步非常重要：不要只把终稿放进项目就结束。把“交接包 / AI 初稿 / 作者终稿 / 你的接受判断”一起交给学习端：

```text
请使用 $novel-revision-learning-loop 做作者改稿复盘。
请对照本章《阶段正文交接包》、AI 初稿、作者终稿和我的接受判断，分析我到底改好了什么，以及 AI 为什么一开始没有做到。

请输出并写回项目：
1. 作者改稿复盘；
2. 总体经验增量：用于之后的策划、蓝图和交接包；
3. 文笔经验增量：用于之后的正文生成；
4. 如果已有多次复盘，请把可复用经验去内容化，分别更新《整体创作经验包》和《文笔执行经验包》。
经验包不能包含这本小说的私有剧情、人名、原句和设定，只保留从本书中提炼出的写法。
```

这样系统会越来越懂你真正想要的“写法”，而不是每一章都从零开始猜。

## 9. 写完一个阶段后必须做什么

每卷虽然提前拆成 3-4 个阶段，但正文写出来后可能出现更自然的情感、意象、角色反应或因果变化。因此系统不要求死守最初的细节计划。

规则是：

- 一个阶段的正文全部验收完成后，默认做一次阶段复盘。
- 如果你让正文端连续生成较多内容，最多在两个相邻阶段完成后必须复盘。
- 已经接受的正文高于旧计划。
- 只校准还没有写出的阶段和章节。
- 如果要改变本卷高潮或全书方向，必须先让你选择。

阶段复盘指令：

```text
当前阶段正文已经完成并逐章验收。请使用 $novel-immersion-engine 在本项目中做《已完成阶段复盘与后续校准》，保存到 `09-阶段复盘与校准` 并更新控制台。

请对照原阶段纲检查：
1. 原定的阶段开始状态、结束状态和主要回报是否真正实现；
2. 人物变化、感情推进、伏笔、关键物品、技能/物品效果、核心代价、信息边界、节奏和声音是否落实；
3. 正文中有哪些有效发现值得正式保留；
4. 哪些内容属于偏移，需要修正；
5. 对尚未写出的后续阶段纲和章节安排有什么影响；
6. 若原路线仍成立，请将校准后的下一阶段纲和下一份正文交接包写回项目；
7. 若影响本卷终点或全书方向，请先提供改线方案让我选择，不要继续生成正文。
```

这一步让系统同时保留两种优点：前期规划的控制力，以及真实正文成长出来的活性。

## 10. 一次写几章最合适

不要为了省事让正文端一次写太长。推荐如下：

| 使用阶段 | 推荐正文生成量 | 原因 |
| --- | --- | --- |
| 第一次测试正文端 | 1 章 | 先确认它是否理解蓝图、口吻是否合适 |
| 流程稳定以后 | 2-3 章 | 能保持连贯，也方便逐章验收 |
| 同一阶段很清楚时 | 一个完整阶段 | 可以看阶段推进效果，但写完立即复盘 |
| 超过两个阶段 | 不推荐 | 偏移累积后，修改代价会变大 |

## 11. 工作流会保存哪些资料

实际写一部长篇时，不需要把所有临时讨论都保存下来。工作流创建的每本小说项目已经固定提供以下结构，不要再另建一套散乱文件：

```text
小说项目/
  00-项目运行规则.md
  00-项目控制台.md
  01-输入素材/
  02-小说圣经与总纲/
  03-分卷与阶段规划/
  04-当前故事状态/
  05-正文交接包/
  06-待验收正文/
  07-已接受正文/
  08-验收与状态增量/
  09-阶段复盘与校准/
  10-版本对比实验/
  11-作者改稿学习与经验包/
```

| 文件 | 作用 |
| --- | --- |
| 项目运行规则与控制台 | 让技能知道当前节点、下一动作和写回位置 |
| 小说圣经与总纲 | 保持整本书的核心方向和结局重力 |
| 分卷与阶段规划 | 指导连续章节推进并保存内部执行包 |
| 当前故事状态与账本 | 记录已成立事实、弧线、债务、物品和能力边界 |
| 正文交接包 | 保存交给正文端时的依据，方便验收 |
| 待验收正文与已接受正文 | 区分草稿与真正的最高事实来源 |
| 验收与阶段复盘 | 记录正文为何被接受、后面的计划为何变化 |
| 作者改稿学习与经验包 | 记录作者怎么改、为什么改，并把经验拆成整体创作经验与文笔执行经验 |

注意：候选方案、未选择的方向和正文端随手添加的设定，不能自动写入“当前故事状态”。

## 12. 第一次试跑推荐方案

不要一开始就规划整本书的所有章节。第一次验证这套系统时，建议跑一个小闭环：

```text
1. 通过工作流创建小说项目，并把想法放入 `01-输入素材`。
2. 使用 $novel-immersion-engine 接管项目，生成小说圣经和 3 个总纲方向并写回项目。
3. 你选择总纲。
4. 策划端生成第一卷分卷纲。
5. 策划端把第一卷拆成 3-4 个阶段。
6. 只选择第一阶段，生成约 3-5 章的章节纲。
7. 先为第 1 章生成正文交接包。
8. GPT 或 Claude 使用 `$novel-prose-renderer` 或正文执行提示写第 1 章正文。
9. 你直接接受，或亲手改成更满意的作者终稿。
10. 如果你改过正文，使用 `$novel-revision-learning-loop` 复盘改稿，并更新总体经验与文笔经验。
11. 把正文放入 `06-待验收正文`，由策划端验收并写回状态。
12. 流程稳定后，完成第一阶段余下正文。
13. 策划端做阶段复盘并校准第二阶段。
```

完成这一轮后，你重点判断：

- 生成的正文是否比直接叫 GPT 写更稳定、更有目的。
- 蓝图是否太重，哪些栏目真正帮助了正文。
- 正文端是否在开放创造位中写出了蓝图没有预设、却让作品变好的东西。
- 人物和感情变化是否自然落进场景，而不是停留在规划文字里。
- 阶段复盘是否发现了原计划没想到、但值得保留的东西。

## 13. 常见错误

| 错误做法 | 会出现的问题 | 正确做法 |
| --- | --- | --- |
| 有一个想法就直接写正文 | 前几章漂亮，后面没有方向 | 先选总纲，再进入当前卷和阶段 |
| 只给正文端单章提纲 | 本章能写，但前后推进断掉 | 一并提供紧凑的阶段导航 |
| 把所有资料全塞给正文端 | 模型混乱，文笔被后台信息压死 | 只发阶段导航、蓝图和必要短片段 |
| 正文一生成就继续下一章 | 意外设定和偏移不断累积 | 逐章验收，通过后再入账 |
| 写完整卷才回头检查 | 路线偏了以后很难修 | 一个阶段完成后就复盘 |
| 规划锁得过死 | 正文像填空，正确却无趣 | 锁住固定锚点，明确开放创造位，让正文端发明过程 |
| 把角色当固定标签 | 人物不会成长 | 用压力、选择和代价逐步改变人物 |

## 14. 最短可复制启动指令

当你只想立刻开始，不想看完整流程时，把下面内容发给 Codex 或 Claude Code：

```text
请使用 $novel-immersion-engine 接管本小说项目。先读取《00-项目运行规则.md》《00-项目控制台.md》和当前节点需要的文件。你不默认写正文，而是负责先把故事推演清楚，把正式产物写回对应目录，并在正文回来后验收和校准后续路线。

如果项目接入的是初始想法：先形成小说圣经和 3 个小说总纲方向供我选择。
如果项目接入的是已有大纲：保留核心，诊断问题并提出增强方案。
如果项目接入的是已有正文：先反向提取真实状态和当前所在阶段，再规划续写。

路线确定后，按“小说总纲 -> 分卷纲 -> 每卷 3-4 个阶段纲 -> 章节纲 -> 阶段正文交接包”的顺序推进。需要正文时，将交接包写入 `05-正文交接包`，让我交给 GPT 或 Claude。交接包要锁住固定锚点和禁止越界事项，同时标明边界内创造与开放创造位，不能把正文变成机械填空。

正文返回后，你逐章验收并写回项目。一个阶段完成，或最多两个相邻阶段完成后，你必须进行阶段复盘，只调整还没写出的后续计划；若影响本卷高潮或整本书方向，先给我方案选择。
```
