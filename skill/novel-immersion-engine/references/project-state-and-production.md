# Project State And Production

Use this reference for any novel expected to span multiple sessions, many chapters, multiple outline alternatives, or continuation from an existing manuscript.

## Authoritative Current-Story State

Maintain one compact master record for the novel. Treat it as the only approved summary of the current canon and future route.

- ...

- ...

Rules:

- Store only accepted facts and chosen future direction here.
- Track where the project currently sits in the five-level hierarchy from `hierarchical-outline-system.md`.
- Cite the chapter or user decision that supports important facts.
- Keep discarded variants and possible ideas out of confirmed canon.
- If new prose conflicts with this state, surface the conflict before updating the state.
- Maintain long-term `Author Intent`, short-term `Current Focus`, authority order, hook debt, cadence, resources, and subplots using `serialization-governance.md` when the project needs them.

## Variant Status Register

Every alternative plan begins as a candidate. Nothing enters canon simply because it appeared in an option.

```markdown
| Variant ID | Scope | Core change | Status | Selected parts | Rejected parts | User decision/source | Canon impact |
| --- | --- | --- | --- | --- | --- | --- | --- |
```

Allowed statuses:

- `candidate`: available for comparison only.
- `selected`: chosen as current future plan, but not yet established by prose.
- `merged`: selected pieces incorporated into another plan.
- `rejected`: do not reuse as fact or setup unless newly proposed and approved.
- `canonized`: realized in accepted prose or explicitly declared canon by the user.

Rule:

```markdown
Only canonized facts may be treated as past story truth. Selected future beats guide drafting but may still be revised before acceptance.
```

## Large Manuscript Ingestion

Do not attempt to reverse-outline a long manuscript from memory or one oversized pass.

Workflow:

1. Split existing prose into manageable chapter batches, usually 5-10 chapters.
2. For each batch, produce a source-linked packet.
3. Consolidate packets into the authoritative current-story state.
4. Identify contradictions, drifting plans, missing text, and unresolved promises.
5. Only then generate future outline variants.

Batch packet:

```markdown
## Batch State Packet
- Chapters processed:
- Source completeness: complete / partial / summary only

### Chapter Reverse Outline
| Chapter | Events | Turn/change | Character movement | Relationship movement | Setup/payoff | Key objects | Abilities/effects | Core cost | Ending pull |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

### Facts Extracted
| Fact | Source chapter | Confidence: explicit/implied/uncertain | Canon conflict? |
| --- | --- | --- | --- |

### Open Threads
| Thread | Last source chapter | Current expectation | Risk if forgotten |
| --- | --- | --- | --- |

### State Changes To Merge
- ...
```

Never pretend unprocessed chapters have been checked. Record the gap.

## Returned Prose Acceptance Gate

A chapter generated by GPT, Claude, or an optional trial draft is a proposal until it passes this gate or the user accepts it.

```markdown
## Draft Acceptance Check
- Draft chapter:
- Blueprint used:
- Chapter execution packet checked:
- Contract followed:
- Planned payoff realized:
- Planned setup/payoff changes:
- Promise debt movement:
- Character/relationship movement realized:
- Core cost movement:
- Key object movements:
- Ability/item-effect use, limit, cost, unlock, counter, or changed rule:
- Countable resource movements:
- Subplot movements:
- Cadence effect or repetition risk:
- Current focus served:
- Knowledge boundary violations:
- Voice drift:
- Creative-space realization: inventive and valid / flat execution / overstepped boundary
- New discoveries introduced by draft:
- Contradictions with current-story state:

## Decision
- Accept as canon:
- Revise before canon:
- Keep as optional discovery:
- Reject:

## State Updates After Acceptance
- ...
```

Treat good expressive improvisations and bounded scene inventions as possible keeper details rather than violations merely because the blueprint did not pre-script them. Treat any improvisation that changes plot, reveal, character, relationship, object, ability/item-effect, resource, promise, or core cost as a proposed discovery requiring approval. Do not silently rewrite the future around it.

## Completed Stage Checkpoint

Run this after all chapters in one drafted stage have passed acceptance, or after two adjacent drafted stages at the latest. It bridges generated prose back into planning: accepted prose becomes authority, while unwritten stage outlines remain adjustable.

```markdown
## Completed Stage Checkpoint
- Volume/stage reviewed:
- Accepted chapters covered:
- Original planned entry/exit state:
- Actual accepted entry/exit state:
- Stage completion status: achieved / achieved with useful discoveries / needs local repair / route decision required

### Stage Outcome Comparison
| Track | Planned result | Accepted prose result | Gap or discovery | Action |
| --- | --- | --- | --- | --- |
| Plot and information | | | | |
| Character and relationship | | | | |
| Promises/setups | | | | |
| Core cost/objects/abilities/resources | | | | |
| Cadence/voice | | | | |

### Future Route Impact
| Unwritten stage or payoff | Still valid? | Required adjustment | Why | User decision needed? |
| --- | --- | --- | --- | --- |

### Next Planning Output
- Canon/state updates settled:
- Current-volume route: preserve / recalibrate / present reroute variants
- Next stage inheritance:
- Next handoff to produce:
```

Rules:

- Complete individual chapter acceptance before treating a stage as completed.
- Stage checkpoints adjust only unwritten plans unless the user explicitly requests a revision or retcon of accepted prose.
- Do not continue drafting automatically if an accepted discovery breaks a volume endpoint, major payoff, character causality, or selected novel direction; offer repairs or reroute choices first.
- Use a stage checkpoint to prevent a carefully planned volume from becoming rigid when real prose discovers a stronger local path.

## State Delta And Degraded-State Protocol

After an accepted chapter, record only changes caused or confirmed by that chapter. Do not rewrite the whole state from memory.

```markdown
## Accepted Chapter State Delta
- Chapter:
- Accepted by/source:

### Canon Added Or Clarified
| Fact | Evidence in chapter/user decision | Confidence | Consequence |
| --- | --- | --- | --- |

### State Movements
| Track | Before | Delta | After | Write-back ledger |
| --- | --- | --- | --- | --- |

### Promise, Cadence, Ability, Resource, Subplot Write-Back
- Promises opened/advanced/deferred/resolved/retired:
- Cadence trail entry:
- Ability/item-effect deltas:
- Resource deltas:
- Subplot deltas:
- Next chapter must inherit:
```

If an extracted change conflicts with accepted prose, existing canon, or an unverified portion of the manuscript:

```markdown
## Pending Verification / Degraded State
| Disputed item | Draft claim | Existing authority | Risk if adopted | Needed confirmation or repair |
| --- | --- | --- | --- | --- |
```

Rules:

- Preserve the prior confirmed state until the disputed item is repaired or accepted.
- Do not continue automatically from an uncertain fact that controls motivation, identity, knowledge, key objects, abilities/item effects, resources, or payoff.
- It is acceptable to mark the current chapter as accepted while keeping one proposed discovery pending verification.

## Key Object Ledger

Track an object when it can affect action, reveal identity or information, carry a relationship, stage a payoff, or accumulate changed meaning.

Object classes:

- **Functional**: weapon, medication, key, tool, device, evidence, resource.
- **Emotional**: gift, relic, photograph, notebook, clothing, keepsake.
- **Structural**: setup/payoff carrier, identity proof, rule demonstration, thematic echo.

An object may occupy more than one class.

```markdown
## Key Object
- Object:
- Class:
- Origin:
- Original owner:
- Current holder/location:
- Physical condition:
- Practical function:
- Emotional meaning:
- Structural/foreshadowing function:
- Who knows it exists:
- Who knows its true meaning:
- Who misunderstands it:
- Planned transformation/payoff:

## Object Movement
| Chapter/scene | Holder/location before | Action/change | Holder/location after | Who witnessed/learned | Meaning changed? | Consequence |
| --- | --- | --- | --- | --- | --- | --- |
```

Rules:

- Do not let significant objects teleport, vanish, or acquire meaning without setup.
- Track transfer, concealment, damage, destruction, recovery, gifting, theft, return, and discovery.
- For emotionally important objects, track how the relationship to the object changes, not merely where it is.
- Do not treat an autonomous living or sentient being as an object; track it under character, relationship, witness, or world-state records as appropriate.

## Ability And Item-Effect State

Track executable powers separately from an object's custody and symbolic meaning. Use this for character skills, techniques, supernatural abilities, equipment effects, unlocks, upgrades, charges, cooldowns, counters, or hidden rules.

```markdown
## Ability/Item Effect
- Ability or item effect:
- User/holder:
- Acquisition or source:
- Current mastered scope:
- Activation condition:
- Demonstrable effect:
- Hard limit:
- Cost or side effect:
- Counter or failure mode:
- Who knows or misreads it:
- Next planned showcase, upgrade, reveal, or payoff:

## Use And Change Log
| Chapter/scene | Use or discovery | Setup already established | Cost/constraint applied | State after | Consequence | Accepted? |
| --- | --- | --- | --- | --- | --- | --- |
```

Rules:

- Do not permit a decisive capability to materialize at the moment it is needed without prior source or setup.
- A clever use may be invented within the established boundary; a new effect, upgrade, counter or cost is a proposed story-state change.
- Keep ability progression connected to reader reward and consequence, not only larger numbers.
