# 工作流驾驶与持续小说项目操作

当用户使用一个带有 `00-项目控制台.md` 的小说项目文件夹，或要求在 `小说沉入工程-工作流` 中推进项目时，使用本规则。

## 驾驶原则

本技能是工作流的策划驾驶引擎，不另建一份平行的聊天版项目状态。

- 若同时存在 `00-项目运行规则.md` 与 `00-项目控制台.md`，视为一个可接管的正式项目；先读两者，再工作。
- 对正式项目生成的总纲、规划、交接包、验收记录、状态增量与复盘结论，必须写入约定目录；聊天回复只说明做了什么、作者需决定什么和下一步是什么。
- 若用户要长期经营一本小说，但当前没有项目文件夹，优先新建或引导新建工作流项目，再进入正式规划。
- 若用户只要讨论、评估或临时试一章，不强迫建项目；这些输出是候选材料，除非随后被作者纳入项目。

## 工作流与技能的分工

- 技能负责驱动工作流并作出推理判断：五级推演、创造自由、正文验收、阶段复盘和路线校准。
- 项目文件夹负责持续状态：当前处于哪一步、文件放在哪里、启用了哪些账本、什么已选定、什么仍待作者决定。
- 新建项目含 `00-项目运行规则.md`，它是该项目可移交给其他 AI 时的独立操作契约；先读它，再读控制台。
- `00-项目控制台.md` 是导航首页，不是 canon 本身。
- 已接受正文和 `04-当前故事状态/当前故事状态.md` 才承载已经确认的故事事实。
- `11-作者改稿学习与经验包` 承载作者改稿复盘和两类经验包；经验包是方法，不是剧情事实。

## 约定目录

| 目录 | 写入内容 |
| --- | --- |
| `01-输入素材` | 用户给出的想法、大纲或已有正文 |
| `02-小说圣经与总纲` | 小说圣经、总纲候选、已选总纲 |
| `03-分卷与阶段规划` | 分卷纲、当前卷阶段纲、当前阶段章节纲、内部章节执行包与场景卡 |
| `04-当前故事状态` | 仅确认事实、权威状态、账本启用清单与按需账本，包括需要时的能力/物品效果规则 |
| `05-正文交接包` | 发送给 GPT / Claude 的阶段导航与章节写作蓝图 |
| `06-待验收正文` | 正文模型返回、尚未确认的稿件 |
| `07-已接受正文` | 已通过验收或作者明确接受的正文 |
| `08-验收与状态增量` | 逐章验收结论与写回变化 |
| `09-阶段复盘与校准` | 阶段检查、后续校准、卷末复核与下卷继承 |
| `10-版本对比实验` | 同一任务下不同蓝图版本的比较 |
| `11-作者改稿学习与经验包` | 作者改稿复盘、总体经验增量、文笔经验增量、整体创作经验包、文笔执行经验包 |

## 每次接手项目的顺序

1. 确认项目根目录，同时找到 `00-项目运行规则.md` 与 `00-项目控制台.md`；若缺一项，先判断这是旧项目待迁移，还是用户仅在做临时任务。
2. 读取 `00-项目运行规则.md` 与 `00-项目控制台.md`，确认当前节点、下一动作、启用账本和待作者决定事项。
3. 根据当前任务读取最少必要材料：
   - 总纲/分卷规划：读取输入素材、已产生的圣经或已选总纲。
   - 阶段与交接包：读取已选总纲、分卷/阶段规划、当前故事状态及涉及的启用账本。
   - 正文验收：读取对应交接包、内部执行包/场景卡、待验收正文、当前故事状态及涉及账本。
   - 阶段复盘或卷末复核：读取相关规划、交接包、已接受正文、逐章验收、当前状态与涉及账本。
   - 作者改稿学习：读取对应交接包、AI 初稿、作者终稿、验收记录和现有经验包。
   - 经验打包：读取多次经验增量，输出去内容化的整体创作经验包与文笔执行经验包。
4. 生成本步产物并放入约定目录。优先采用可识别的中文文件名，如 `第1卷-阶段1-第03章-交接包.md`。不要以聊天文本代替应当存在的正式文件。
5. 只有发生以下事项时写回权威状态：作者选择路线、作者确认设定、正文验收接受、阶段复盘确认后续校准。
6. 更新控制台中的当前节点、最新产物、作者待决定事项和下一步动作。
7. 向作者简短汇报已落盘文件、需要选择的事项和后续入口，不重复倾倒整份后台内容。

## 尚无项目时

当用户提出新书、已有大纲改造或长篇续写，并准备持续使用时：

1. 先建立工作流项目，接入类型选择为 `初始想法`、`已有大纲` 或 `已有正文`。
2. 将素材保存或引导放入 `01-输入素材`。
3. 从 S0 项目接入开始，先形成基础状态和需由作者选择的问题。
4. 在项目未建立前生成的讨论性内容，只能作为候选输入，不得假装已是项目 canon 或正式已选总纲。

## 节点通过条件

| 节点 | 只有满足以下条件才向后推进 |
| --- | --- |
| 小说圣经与总纲 | 核心读者承诺、故事发动机、结局重力和选定路线已确认，候选与正式路线分离 |
| 分卷与阶段规划 | 当前卷有独立回报和终点；当前阶段有清楚的入口状态、出口状态、因果链与章节骨架 |
| 正文交接 | 已先完成需要的内部执行包/场景卡，固定锚点、创造空间、知识边界、能力/物品效果边界和相关状态变化已解决 |
| 正文验收 | 只有接受正文才写入 canon 和账本；待改正文不建立事实 |
| 阶段复盘 | 阶段内章节均已验收，已决定哪些发现吸收、哪些偏移修复，并复核未写阶段 |
| 卷末复核 | 本卷承诺与欠债已结算，下一卷必须继承的状态已经明确 |
| 作者改稿学习 | 蓝图、AI 初稿、作者终稿已对齐，复盘拆出总体经验增量与文笔经验增量 |
| 经验打包 | 经验增量完成去内容化，适用范围、置信度、使用方式和边界清楚 |

## 状态保护规则

- 候选总纲、候选阶段和正文草稿不是已确认事实。
- 正文被接受前，不因其中的新设定修改未来路线。
- 已接受正文与旧规划冲突时，保留正文，调整尚未写出的规划或向作者提出改线选择。
- 控制台只记录当前流程状态和文件索引；不要让它膨胀成完整后台账本。
- 账本按 `04-当前故事状态/账本启用清单.md` 启用；不影响本项目的模块保持未启用，避免机械填表。
- 作者改稿复盘可以保留具体内容，但经验包必须去内容化；不要把本书私有剧情误放进通用经验。

## 正文交接与创造空间

输出交接包时：

- 整章、续写、有关键选择/揭示/代价/物品/技能或装备效果/关系转折的工作，使用 `05-正文交接包/完整交接包模板.md`；仅低风险短场景或试写可采用快速模板。
- 先在策划端形成内部章节执行包与场景卡，解决长线判断，再把正文真正需要的责任提炼成交接包。
- 写清固定锚点、信息边界、关键状态变化与结尾落点。
- 写清边界内创造，让正文端自行设计场景抵达方式。
- 尽量为每章保留开放创造位，仅规定需要产生的感受或戏剧功能。
- 不把完整后台资料倾倒给正文端。
- 若有 `文笔执行经验包`，只把适用的文笔执行经验提供给正文端；它指导表达，不授权正文端改剧情。

验收正文时：

- 既检查是否违背固定锚点，也检查开放创造是否带来有效惊喜。
- 不把合法的局部创造误判为越界。
- 会改变长线的发现必须交作者决定，或在阶段复盘中明确提出。
- 接受后，把确认过的变化写回当前状态与已启用账本；尚有争议的发现保持待确认。

## 更新控制台的最低内容

完成一次工作后，至少更新：

- `当前节点`
- `当前卷/阶段/正文范围` 中发生变化的项目
- 状态看板中的本节点状态和最新产物
- `当前需要作者决定`
- `下一步动作`

不要在用户只是讨论方向、尚未要求落盘时擅自修改项目文件。用户已经要求继续工作流或生成产物时，直接推进并写回。
